# Sangfor XDR OctoBus Service

OctoBus service package for Sangfor XDR OpenAPI, covering asset management, security incidents, linkage response, vulnerability management, SOAR, OAuth2 auth, and threat expert incidents. 83 API endpoints mapped to 90 gRPC methods across 7 services.

Service root: `services/sangfor__xdr`.

## Import

```bash
octobus service import --id sangfor-xdr ./services/sangfor__xdr
```

## Configuration

```json
{
  "xdrBaseUrl": "https://xdr.example.com",
  "timeoutMs": 10000,
  "skipTlsVerify": false
}
```

Config alias: `endpoint`, `restBaseUrl`, `baseUrl` are accepted as `xdrBaseUrl` alternatives.

## Secrets

```json
{
  "accessKey": "your-xdr-access-key",
  "secretKey": "your-xdr-secret-key"
}
```

Secret aliases: `ak`, `sk`.

## Authentication

AK/SK HMAC-SHA256 signing per Sangfor OpenAPI spec (ported from Go SDK `aksk/v3`). Each request includes `Authorization`, `sdk-content-type`, `sdk-host`, and `sign-date` headers auto-generated by `src/sangfor-sign.js`.

## Error Mapping

- `INVALID_ARGUMENT`: missing required fields, invalid parameter values
- `PERMISSION_DENIED`: HTTP 401/403 from XDR upstream
- `UNAVAILABLE`: HTTP 5xx, network errors, DNS failures
- `UNKNOWN`: unexpected response format or JSON parse failures

## Behavior Notes

- All requests use AK/SK HMAC-SHA256 signing via `src/sangfor-sign.js`.
- The service runs in `long-running` mode for connection reuse.
- Write operations (BanIP, IsolateHost, etc.) are idempotent where the upstream supports it.
- Response fields are mapped with fallback aliases to handle API version differences.

## Supported Version

Sangfor XDR platform with OpenAPI v3 (AK/SK authentication). Tested with API paths from eolink API documentation.
